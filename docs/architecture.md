# 架构设计

本文描述喜马拉雅音频下载器（Ximalaya-Downloader-Next）的软件架构、模块划分与关键设计决策。项目背景与使用方式见 [`overview.md`](./overview.md)。

## 1. 概览

项目采用**六边形架构（端口与适配器）+ 任务引擎**：

- 一个**纯领域核心**，只包含业务概念与规则，不做任何 I/O。
- 一组**端口（抽象接口）**，描述核心需要的外部能力。
- 一组**适配器**，是端口的具体实现（网络、存储、解码、签名等）。
- 一个**任务引擎**，负责把下载工作编排成可调度、可恢复的流程。
- 一个**装配根**，在启动时按配置把适配器接到端口上，并组装出供前端调用的门面。

这样划分的目的，是把「与平台强相关、可能频繁变动」的部分（鉴权签名、媒体解码、接口地址等）全部隔离在端口之后。它们变动时只需替换对应的适配器或更新配置，核心逻辑、任务编排与前端都不受影响。

## 2. 设计目标与原则

1. **核心库优先**：能力沉淀在核心库，前端（CLI、未来的 GUI/Web/API）只依赖门面，互不耦合。
2. **隔离易变点**：把最可能变动的部分关在窄接口后面，更新影响面最小化。
3. **稳定可靠**：状态持久化 + 断点续传 + 错误分级重试，长时间/大批量运行可控、可恢复。
4. **可扩展**：新增音源、输出方式、签名实现或前端，都通过实现既有端口完成，而非修改核心。
5. **可测试**：领域是纯逻辑；所有外部依赖都是可替换的端口，便于用替身进行单元与契约测试。

## 3. 分层架构

```
┌──────────────────────────────────────────────────────────┐
│  前端(薄壳)   CLI(当前)   │  GUI / Web / HTTP-API(未来)   │
└───────────────┬──────────────────────────────────────────┘
                │ 仅依赖门面
┌───────────────▼──────────────────────────────────────────┐
│  应用层  Facade(门面) + 用例 + 任务引擎(调度/重试/限速     │
└───────────────┬───────────────────────────┬──────────────┘
                │ 调用                        │ 依赖
┌───────────────▼──────────────┐   ┌────────▼──────────────┐
│  领域核心(纯，无 I/O)         │   │  端口(抽象接口)        │
│  实体 / 值对象 / 状态机       │   │  签名/网络/音源/解码/  │
│                              │   │  输出/存储/限速/...    │
└──────────────────────────────┘   └────────▲──────────────┘
                ┌─────────────────────────────┘ 实现
┌───────────────▼──────────────────────────────────────────┐
│  适配器层  签名 / 网络 / 在线音源 / 媒体解码 /             │
│            文件输出 / SQLite 存储 / 凭据存储              │
└──────────────────────────────────────────────────────────┘
        ▲ 装配根按配置注入实现并组装门面
```

依赖方向始终**由外向内**：前端依赖应用层，应用层依赖领域与端口，适配器实现端口。领域核心不依赖任何外层。

## 4. 目录结构

```
src/xdl/
├─ domain/            实体 / 值对象 / 状态机 / 命名策略（纯，无 I/O）
├─ ports/             抽象接口（Protocol）
├─ application/
│  ├─ usecases/       解析单曲 / 解析专辑 / 下载 / 专辑同步 / 登录 / 本地解码
│  ├─ engine/         任务引擎：调度 / 状态机驱动 / 重试 / 限速
│  └─ facade.py       对外门面（库的公开 API）
├─ adapters/
│  ├─ sign/           签名实现（默认实现 + 兜底实现 + 降级组合）
│  ├─ http/           网络传输
│  ├─ source/         在线音源（未来可加更多音源）
│  ├─ decode/         媒体解码 / 本地文件解码
│  ├─ sink/           文件输出 + 后处理（标签、转码…）
│  └─ store/          SQLite 任务库 / 凭据存储
├─ config/            数据化配置（接口地址、解码参数等，随版本更新）
├─ settings.py        用户配置加载与校验
├─ errors.py          类型化异常体系
├─ composition.py     装配根（依赖注入 + 降级链）
└─ frontends/cli/     命令行前端
```

把模块拆小、各自职责单一，是为了让每个单元都能独立理解、独立测试、独立替换。

## 5. 领域模型

领域核心是纯对象，不触网络、不读写磁盘，因此快速且易测。

**主要实体与值对象**

- `Quality`：音质枚举，含「降级协商」——请求的音质不可用时按规则回退到次优。
- `PlayUrl`：单个可播放资源（音质、格式、地址、体积）。
- `Track` / `Album`：音频与专辑，含标题、序号、时长、是否付费、是否已授权等。
- `DownloadTask`：任务引擎的工作单元（见状态机）。
- `AuthSession`：登录会话（用户、会员状态、凭据等）。
- `NamingPolicy`：文件命名策略（非法字符过滤、序号补零、命名模板），放在领域层以保证各前端命名一致。

**`DownloadTask` 状态机**（断点续传与重试的基石，状态转移显式）

```
PENDING → RESOLVING → RESOLVED → DOWNLOADING → COMPLETED
   │           │           │            │
   │           └─ FAILED(可重试 / 永久) ─┘
   └─ SKIPPED(已存在 / 未授权)                 ↕ PAUSED
```

## 6. 端口与适配器

端口是核心对外部能力的抽象声明，适配器是其实现。下面是主要端口及其职责：

| 端口 | 职责 |
|---|---|
| `SignProvider` | 为受保护接口的请求生成所需鉴权签名 |
| `HttpTransport` | 发起 HTTP 请求与流式下载（支持断点 Range） |
| `Source` | 音源：获取单曲、专辑曲目、专辑信息、搜索 |
| `Decoder` | 将平台返回的音频资源解码为可用形式 |
| `MediaSink` | 输出：落盘、断点偏移、后处理（标签/转码） |
| `TaskStore` | 任务与历史的持久化（含断点与增量游标） |
| `CookieJar` | 登录会话与凭据的读写持久化 |
| `RateLimiter` | 请求速率控制（规划端口，MVP 尚未独立实现） |
| `ProgressReporter` | 向前端回报进度（由前端实现） |
| `HookBus` | 扩展事件总线（请求前 / 下载后 / 出错） |
| `Clock` | 时间源（便于测试） |

**端口粒度原则**：不盲目细拆。一个能力满足以下任一条才独立成端口——① 变化原因/频率与邻居不同；② 确实会有第二种实现；③ 测试需单独替身。否则合并，避免为不存在的需求过度抽象。例如「专辑增量游标」与任务持久化同源、同变化原因，故并入 `TaskStore` 而不单列。

## 7. 关键子系统

### 7.1 请求鉴权（签名）

访问受保护接口需要为请求附加鉴权签名。该能力抽象为 `SignProvider` 端口，做成**可插拔 + 自动降级**：

- 默认实现：内嵌一个轻量 JS 运行环境（Node）生成签名，**进程长驻、反复复用**，避免重复启动开销；
- 兜底实现：当默认实现不可用或被拒绝时，自动降级到无头浏览器（Playwright），以获得更高的环境保真度；
- 签名中较稳定的部分会缓存到本地、跨启动复用，易失效的部分按需刷新。

降级与健康度由一个组合实现统一处理，用例层只依赖 `SignProvider` 抽象，不感知具体用了哪种实现，未来也可新增实现接到链上。无头浏览器属较重依赖，按需安装而非强制打包。

> **MVP 现状（与上文设想的偏差）**：当前未单独实现 `SignProvider`，而是采用「**让页面自己签名**」的方案——`ChromeSource` 加载已登录页面，由页面内的 `du_web_sdk` 自动生成 `xm-sign` 并发出 `baseInfo` 请求，适配器通过 Playwright `response` 事件只读提取目标响应。早期版本会注入脚本改写 `XMLHttpRequest.prototype.open/send`；实测这会暴露明显的非原生函数指纹，现已删除。签名逻辑仍隐含在音源适配器内。
>
> **反自动化风控（重要）**：历史测试中，自己启动真实 Chrome 再用 CDP 接管比 Playwright `launch` 更可靠，但 2026-07-11 的 Chrome 150 复测已经推翻“CDP 真实 Chrome 稳定过风控”的结论：即使有头，XDL 页面也弹出大量验证码并返回 `3005`，而用户日常浏览器快速播放正常。当前可疑差异包括远程调试/CDP、专用 Profile 信誉、每曲新建页面，以及单页自动加载大量推荐 `baseInfo`。因此该适配器仍属实验性实现，不能以 UA/stealth 等伪装作为解决方案。
>
> **2026-07-11 设备指纹重置失效、CDP 被锁定为根因**：进一步实测显示，重置设备 Cookie（`_xmLog`/`wfp`/`Hm_lvt_*`）甚至连 `localStorage`/`sessionStorage`/`IndexedDB` 一起清掉，每次仍只能下载 3 集就被服务端判罚（返回 `1001/系统繁忙`，连验证码恢复路径都不给）。逐项排除后唯一剩下的差异是 **`--remote-debugging-port` + Playwright 通过 CDP 接管** 本身——`du_web_sdk` 能从 inspector 痕迹精确识别 CDP 环境，3 集后累计扣分到阈值。详见 `docs/risk-control-observations.md`。由此"指纹伪装"这条路在 Playwright/CDP 框架内走到头；下一步治本方向是**浏览器扩展 + 本地原生消息（Native Messaging）**：扩展监听 `baseInfo` 响应经 `chrome.runtime.connectNative` 回传 XDL，XDL 完全不用 CDP。`ChromeSource` 作为失败时的备用通道保留，未来由 `ExtensionSource` 实现 `Source` 端口替代它。
>
> **并发与风控**：`ChromeSource` 仍支持在共享上下文里开独立 page，但受保护的`baseInfo` 默认串行（`max_concurrency=1`）。2026-07-11 复测中，单次解析成功，随后 K=4 的 8 次解析仅 1 次成功、7 次返回 `3005/系统繁忙`；35 秒后单次重试仍未恢复。因而旧的“K≤6 安全”结论已作废。首个 `1001` 或语义为“系统繁忙”的 `3005` 会被分类为 `RiskControlError` 并熔断整批，不进入自动失败收尾轮。结果写入最小化 JSONL 供离线观测。公开门面保持同步签名，内部 `asyncio.run` 收口。

### 7.2 在线音源

规划中的 `XimalayaWebSource` 会实现 `Source` 并经 `HttpTransport` 与 `RateLimiter`
发出请求；MVP 当前实际使用 `ChromeSource`，由真实页面发出受保护请求，代码尚无独立
`RateLimiter`。音质在领域层协商，请求音质缺失时回退。

架构允许未来挂接更多音源（都实现同一 `Source` 端口，可作为彼此的备用来源）。

> **下一步：`ExtensionSource`**：`ChromeSource` 的风控根因是 CDP 接管本身，不可在 Playwright/CDP 框架内消除（见 §7.1 反自动化风控备注）。下一步新增 `ExtensionSource` 作为 `Source` 端口的主实现：用户在自己的 Chrome（不启动 `--remote-debugging-port`）安装 XDL 扩展，扩展用 `chrome.webRequest.onCompleted` 监听 `baseInfo` 响应并经 `chrome.runtime.connectNative` 回传；XDL 进程跑一个 native messaging host 接收并加入任务库，下载走 `requests`。`du_web_sdk` 看到的执行环境与用户日常浏览器完全一致。`ChromeSource` 仍保留作为扩展不可用时的备用通道。

> **专辑清单的取法（MVP 现状）**：`get_album` 走平台**免签的「非 v1」**接口 `/revision/album/getTracksList`（纯 HTTP 翻页，每页 30 条，匿名可取，含曲目 id/标题/序号），因此专辑清单不依赖浏览器与登录。注意其 `/v1/` 版本反而要 `webtk` 签名，且对自动化浏览器做环境风控（返回「当前环境异常，请使用正规浏览器」），故弃用。逐集 `playUrl` 仍走 §7.1 的baseInfo 截获——专辑下载时**复用同一长驻浏览器**逐集导航解析，避免每集冷启动。

### 7.3 媒体解码

平台返回的部分音频资源在使用前需要一步解码。该逻辑封装在 `Decoder` 端口之后，与网络和业务逻辑解耦。把它独立成端口的原因是：它是最可能随平台变化而需要更新的部分；隔离后，更新时只动这一个适配器并配合「金标向量」测试回归即可，不波及其他模块。

### 7.4 本地文件通道（备用获取方式）

除在线获取外，架构预留 `XmFileSource`，用于把平台客户端导出的本地音频文件作为备用来源进行解码。它实现统一的 `Source` 端口，因此能复用同一套下载/解码流水线，自动获得进度、去重与重试能力。

### 7.5 稳健的请求策略

MVP 当前采用串行默认值、错误分级退避和风控熔断。批次 worker 启动前只有 0–0.3 秒
轻微错峰，尚不等同于全局 RateLimiter；后续需要把受保护接口解析和媒体下载拆成两个
并发域，再实现显式的最小间隔策略。

## 8. 任务引擎与可靠性

任务引擎是「稳定可靠」的核心，位于应用层，靠端口驱动。

### 8.1 状态持久化（SQLite）

```sql
download_task(
  id PK, track_id, album_id, title, quality,
  state,                         -- 状态机当前态
  target_path, part_path,        -- 最终路径 / .part 临时路径
  total_bytes, bytes_done,       -- 字节级续传依据
  attempts, max_attempts, last_error_code, last_error_msg,
  created_at, updated_at, completed_at,
  UNIQUE(track_id, quality)      -- 去重：同曲目同音质只一条
)
album_sync(album_id PK, title, total_known, cursor, last_synced_at)  -- 增量游标
meta(key PK, value)              -- 缓存 / schema 版本 / 迁移
```

`meta.schema_version` + 启动时按版本运行迁移脚本，保证长期演进加字段不丢历史数据。

### 8.2 三层断点续传 + 原子落盘

- **任务级**：未完成的任务下次启动自动被重新调度。
- **字节级**：磁盘上的 `.part` 文件 + 数据库中的 `bytes_done` → 用 HTTP Range 从断点续传；服务器不支持 Range 时整文件重来。
- **专辑级**：`album_sync.cursor` 让重跑只解析新增曲目。
- **崩溃安全**：先写 `.part` → 刷盘 → 完成时原子重命名为最终文件，数据库状态在事务中更新；崩溃后凭 `.part` + 状态续传。

### 8.3 调度：并发 / 限速 / 重试分类

引擎从任务库**分批**拉取任务（内存有界，超大专辑也不会撑爆内存），每个任务走
`解析 → 下载 → 收尾` 流水线，并发由信号量限定。MVP 尚无独立 `RateLimiter`；受保护
请求靠默认串行和首个风控熔断控制。错误按类型分级处理：

```
网络超时 / 5xx        → 指数退避重试（次数受限）
访问被限流 / 鉴权失效  → 冷却 + 刷新鉴权 + 可降级后重试
访问被拒（达到限制）   → 标记失败并给出明确提示
未授权 / 不存在        → 跳过或永久失败，不重试
```

另设「失败收尾轮」统一重试残余失败项，且状态持久化、可跨启动恢复。收到中断信号时优雅停机：完成在途写入、落库、干净退出，下次可续。

### 8.4 专辑增量同步

`SyncAlbum` 用例拉取专辑曲目，与任务库比对（游标 + 去重键），只为新增/变更的曲目建任务，引擎因而只下载新增内容。

## 9. 配置系统

配置分两层，彼此隔离：

```
随代码发布（版本化，更新时一并发版）       用户目录（如 ~/.config/xdl 或 %APPDATA%\xdl）
  config/  接口地址、解码参数等             settings：下载路径 / 默认音质 / 调优参数
        ↑ 与平台相关、随版本更新             secrets：登录凭据等敏感信息
                                            ↑ 用户数据，升级不被覆盖
优先级：命令行参数 > 环境变量 > 用户配置 > 内置默认值
```

启动时在装配根加载为一个**校验过的配置对象**（配置非法则快速失败并给出清晰原因）。可调参数全部可配置，给保守默认值：

| 配置项 | 默认值 | 说明 |
|---|---|---|
| `max_concurrency` | 1 | 受保护播放信息解析/下载流水线并发上限；默认串行 |
| `request_interval` | 尚未实现 | 规划中的全局请求最小间隔；不能用随机错峰替代 |
| `cooldown` | 60 秒 | 受限后的冷却时长 |
| `max_attempts` | 3 | 单任务最大重试次数 |
| `global_retry_rounds` | 2 | 失败收尾轮数 |
| `http_timeout` | 30 / 120 秒 | 解析 / 下载超时 |
| `batch_size` | 200 | 引擎单批任务数 |
| `default_quality` | standard | 默认音质 |
| `download_path` | ./download | 下载根目录 |

与平台相关的配置（`config/`）和用户配置完全隔离，发新版只替换前者，用户的路径、凭据与调优不受影响。

## 10. 横切关注点

### 10.1 错误处理

采用**带分类的类型化异常体系**，引擎按异常类型决定重试策略，而非靠返回值或字符串匹配：

```
XdlError
├─ ConfigError
├─ AuthError(未登录 / 会话过期 / 未授权)
├─ SignError(可重试)
├─ NetworkError(超时 / 连接重置，可重试)
├─ ApiError(限流 / 被拒 / 达到限制 / 不存在 / 响应异常)
├─ DecodeError
└─ StorageError
```

每个异常携带 `retryable` 与 `category`。「预期内的结果」（如未授权）用领域结果对象表达而非抛异常，保持控制流清晰。前端边界兜底未捕获异常：完整堆栈写日志文件，向用户显示友好提示。

### 10.2 日志与可观测

使用标准日志 + 轮转文件（保留历史，不每次清空）+ 控制台美化输出；对凭据等敏感信息**脱敏**；`--debug` 可输出更详细的诊断信息；运行结束打印汇总（下载/跳过/失败/重试/字节数）。

### 10.3 测试策略

六边形架构让测试直接受益——端口都可替身：

- **领域单测**：音质协商、命名、状态机转移等纯逻辑。
- **金标向量**：用「已知输入 → 已知输出」的样本钉死解码等关键算法，配置一旦变动即可发现是否破坏正确性，是安全更新的护栏。
- **适配器契约测试**：同一套测试跑遍某端口的所有实现。
- **网络回放**：录制真实响应、离线回放，解析逻辑的测试不依赖实时网络与凭据。
- **引擎测试**：用内存版任务库与假传输，验证续传、重试、并发与增量。

## 11. 前端与门面

`Facade` 是核心库对外的公开 API（解析、下载、专辑同步、搜索、登录、本地解码等），也是未来其他前端复用的稳定面。当前的命令行前端只是其上一层薄壳：负责参数解析与进度展示（实现 `ProgressReporter`）。未来的 GUI/Web/HTTP API 导入同一个门面即可，核心零改动——这正是「为各种功能预留接口」的落点。

## 12. 扩展点

得益于端口与适配器，常见扩展都不必改动核心：

- **新增音源**：实现 `Source` 端口（如其他获取通道），在装配根注册即可，可作为现有音源的备用。**下一步主实现**：`ExtensionSource`——浏览器扩展 + 本地原生消息（见 §7.2 的"下一步"备注），替代 `ChromeSource` 作为默认在线音源，绕开 CDP 风控根因。
- **新增输出/后处理**：实现或包装 `MediaSink`（如自动写入音频标签、转码）。
- **替换/新增鉴权实现**：实现 `SignProvider`，接入降级链。
- **新增前端**：依赖 `Facade` 门面，实现 `ProgressReporter`。
- **挂接副作用**：通过 `HookBus` 订阅事件（请求前、下载后、出错），无侵入地加入通知、统计等能力。

## 13. 打包与分发

- **开发者**：以 Python 包形式安装（pip / pipx），含库与命令行入口。
- **普通用户**：打包为单文件可执行程序。
- 签名所需的 JS 运行时与资源随包提供；无头浏览器作为兜底依赖，按需下载而非强制打包。
